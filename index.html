01 · How it works

Four steps. Every project. Same shape.

The process is the same whether a project takes a week or a quarter. What changes is depth — not direction.

01

Understand

Capture the client's real intent. Define scope. Mock up the solution. Get sign-off before any code is written.

1–4 weeks · scaled to project size

02

Build

Design the architecture. Build with AI assistance. Tests written before code. Documentation lives alongside.

Most of the project time

03

Ship

Quality checks, security scans, performance tests — all automated. The release only goes live when every check passes.

A few days

04

Run

Production runs with monitoring on by default. Weekly health reports go out automatically. Issues are caught before clients call.

Forever

02 · The team

Three roles. Three team shapes.

Who plays which role flexes with project size. What gets checked along the way never does.

The three roles

PB

Product Builder

Drives requirements, design, and build. Talks to the AI most. Owns the technical drafting from intent to launch.

AR

Architect

Defines guardrails and signs off design. Reviewer and decision-maker, not implementer. Keeps quality consistent across projects.

AM

Account Manager

Owns the client relationship and the intent. Holds scope at kickoff and charter. Drives client sign-offs. Owns go-live handover.

Three team shapes — pick the right size

Solo · 1 person

Builder + AM = same person

One person plays both roles. Architect is shared with the portfolio. DevOps support is shared.

Best for: small features · ~5-day projects

Pair · 2–3 people

Builder + AM separate

Two distinct roles. Architect shared across a few projects. DevOps support shared. Sign-offs are separated cleanly.

Best for: ~2-week projects · customizations

Full team · 4+ people

Every role its own person

Dedicated Architect. Dedicated DevOps engineer for the duration. Multiple Builders possible.

Best for: ~4-week greenfield · government projects · regulated work

03 · What we're building

Five AI assistants. One unified process.

Each role gets a packaged AI assistant — a tested set of prompts, skills, and review rules — so nobody starts from scratch on a new project.

For the Account Manager

Client Conversation Assistant

What it does: Helps the AM capture client intent, run discovery, draft the charter, and keep scope under control through the project.

What stakeholders see: Faster sign-offs. Fewer "that's not what I meant" moments later.

Repos · service-am-plugin · product-am-plugin (one per department)

For the Architect

Architecture Assistant

What it does: Reviews technical designs against Dhwani's reference patterns. Catches drift before it ships.

What stakeholders see: Consistent quality across projects. No rogue architectures.

Repos · service-architect-plugin · product-architect-plugin (one per department)

For Builder + Engineering

Code Building Assistant

What it does: Writes code with AI — but enforces tests-first, documentation-alongside, and proper review before merge.

What stakeholders see: Faster delivery without the quality drops you'd normally expect from speed.

Repos · service-builder-plugin · product-builder-plugin (one per department)

For DevOps + Security

Quality & Delivery Assistant

What it does: Runs security scans, load tests, and the pre-release checklist automatically. Releases ship with a signed report.

What stakeholders see: No surprises in production. Every release has evidence.

Repo · ai-sdlc-devops-plugin (shared)

For DevOps (always-on)

Production Monitoring Assistant

What it does: Watches every live application — uptime, performance, security. Sends weekly health reports automatically.

What stakeholders see: Issues caught before clients call. Monthly reports in their inbox.

Repo · ai-sdlc-devops-plugin (shared)

Shared infrastructure

ai-sdlc-skills

Central library of AI prompts and skills. Every assistant pulls from here. Semver-tagged with built-in tests.

ai-sdlc-hub

Single onboarding site. SOPs, plugin docs, role guides. One URL anyone in the team lands on.

Nine repos total. Each persona-facing assistant (AM, Architect, Builder) ships as two repos — one per department (service-* and product-*) — so each department can tune its plugin without colliding. The DevOps plugin is shared across departments. Plus the two shared infrastructure repos.

04 · What we need

Two engineers. A small budget. Four decisions.

Concrete, time-bound, and reviewable at each quarter boundary.

People

2 dedicated DevOps engineers · 4 quarters.

Engineer 1 · Quality & Delivery anchor

  • Owns Stage C end-to-end (pre-go-live checks)
  • Builds Kotwal CI integration
  • Builds the Release Readiness Report pipeline
  • Owns the ai-sdlc-devops-plugin Quality & Delivery surface

Engineer 2 · Adoption & Operate anchor

  • Builds the Day 1 starter template
  • Builds inter-plugin bridge skills
  • Owns Production Monitoring + the hub
  • Owns the ai-sdlc-devops-plugin Monitoring surface

Plus part-time support from each project's Tech Lead during their rollout window. No new permanent hires beyond what's already planned.

Tooling

Free for most things · small licence budget for the rest.

Free or already-licensed

GitHub (private) · Claude CLI · Mermaid · Google Stitch · Fireflies.ai · Kotwal · Mukhbir · Prometheus · Loki · Tempo · Grafana · Terraform · pre-commit · gitleaks

Needs a small budget

Paid security scanning tier (Semgrep / Snyk) · production status page (Better Uptime or Instatus) · self-service deployment portal — Backstage (OSS, build effort) or paid (Port / Cortex)

Exact annual licence estimate refined in the Q0 inventory phase — before the spend commitment is made.

Decisions

Four standards calls only leadership can make.

  1. Front-end stack — standardise on one, or stay project-dependent?

    Drives which Builder-plugin templates we build first (Vue vs React).

  2. Cloud target — AWS default, with on-premise for government work?

    Drives IaC module choice and Architecture Assistant reference patterns.

  3. Deployment portal — build with Backstage, or buy Port / Cortex?

    Q2 capacity hinges on this — OSS is a build effort, paid is faster but recurring spend.

  4. Project sizing — who classifies solo / pair / full team?

    Determines team-shape assignment per project and DevOps pull on each engagement.

06 · The complete picture

Every stage. Every role. Every gate.

The full AI-SDLC laid out in sequence. Read this once to see how a project moves from a first conversation to a live, monitored application.

01

Intent & Charter

AM (with PB) · using the Client Conversation Assistant

AM-led. Capture the client's real intent. Map stakeholders. Agree the Charter — scope, success criteria, team, escalation.

Gate · Signed Charter

02

Solutioning

PB (AM in client-facing parts) · using the Client Conversation Assistant

Draft functional requirements, user roles, process workflows, data model, wireframes. Mock-ups signed off by client — AM facilitates the client touchpoints.

Gate · Client signs the mock-up

03

Tech Spec & sign-off

PB drafts · Architect signs · using the Architecture Assistant

Tech Spec assembled. Architecture validated against reference patterns. Sign-off captured as a signed git tag.

Gate · Tech Spec signed · ready for the first AI prompt

04

Build & QA loop

Engineering + PB · using the Code Building Assistant

Four AI personas inside one repo: Dev writes code (strictly to docs), QA writes tests first, BA maintains documentation, Reviewer checks PRs. Auto-deploys per branch.

Gate · CI green · review approved · merge to development

05

Client UAT

AM + Client · UAT environment auto-provisioned

Client validates the built product against the original intent. Often optional on small projects — staging can serve as the preview.

Gate · UAT signed · all critical issues resolved

06

Pre-go-live checks

DevOps + Security · using the Quality & Delivery Assistant

Security scans, penetration test, load test, performance baseline, end-to-end test finalisation. All auto-generated. SLOs confirmed.

Gate · Every check green · Release Readiness Report signed

07

Go-Live

DevOps + AM · using the Quality & Delivery Assistant

Production deploys. Post-deploy monitoring tightens for the first 30 minutes. Public users only after the gate passes.

Gate · Stable on production for 30 min · rollback rehearsed

08

Run forever

DevOps · using the Production Monitoring Assistant

Weekly health monitoring. CPU should stay under 50%. 3-tier alert escalation — third alert is critical and triggers auto scale. Weekly + monthly reports to project owners.

Ongoing · feeds learnings back to KM for the next intent

All eight stages happen on every project. Their depth changes with project size — a 5-day mGrant fit-gap moves through them faster than a 4-week greenfield. The gates never get skipped.

08 · From project start to live · the complete lifecycle

Eleven stages. Three iteration loops. One repo end-to-end.

Every trigger event, every environment that spins up, every iteration loop — from the moment a client signs the Charter to the weekly health report two years later.

01

Project starts

Trigger
Charter signed by client + Dhwani PM
Who
AM + PB
What happens
Project classified by department (Service or Product) and team shape (solo / pair / full team). Tech Lead + Architect assigned.
Output
Signed Charter, archived in KM
02

Repo created · one command

Trigger
PB (or AM-as-PB on solo projects) runs the starter command
Who
PB · uses the ai-sdlc-hub starter template
Command
gh repo create dhwani-{dept}-{client}-{project} --template ai-sdlc-stage-a-starter --private --clone
What's pre-wired
Folders for every Stage A artefact · Markdown schemas · CI seed · Pages → CloudFront · .claude/skills/ linked · CODEOWNERS (Architect + AM)
Output
github.com/dhwani-{dept}/{project}
03

Staging auto-provisions

Trigger
First push to repo → GitHub Action fires
Who
Automatic · DevOps plugin handles infrastructure
What happens
Staging env spins up · DNS configured · secrets injected from Vault · first AI prompt ready to execute
SLO
< 15 minutes from repo creation to live URL
Output
{project}-staging.dhwaniris.in live
04

Stage A iterations · Solutioning loop

Trigger
First claude /intent command in the repo
Who
PB drives · AM facilitates client touchpoints · using Client Conversation Assistant ({dept}-am-plugin)
Loop
Each Stage A step (FRs · RBAC · workflows · data model · wireframes) is a PR → review → merge → auto-deploys to staging → client preview → iterate
Typical iterations
5–15 PRs depending on project size
Output
All Stage A artefacts in repo · client-viewable on staging URL
05

Sol Doc signed · ready for first AI-coding prompt

Trigger
All Stage A artefacts complete · Architect reviews
Who
PB drafts Sol Doc · Architect signs · using Architecture Assistant ({dept}-architect-plugin)
What happens
Sol Doc finalised covering all 16 sections. §5.11 Tech Stack + §5.12 Security + §5.16 List of Objects = TRD seeds.
Sign-off
Signed git tag · e.g. v0.1-soldoc · immutable archive
Output
Project moves to "Build" status · first AI-coding prompt can fire
06

Development env spins up

Trigger
First merge to the development branch
Who
Automatic · DevOps plugin provisions from §5.11 Tech Stack
What happens
Dev env auto-provisions via Terraform · CI pipelines live · auto-deploy on every development push
Output
{project}-dev.dhwaniris.in live
07

Stage B iterations · Build & QA loop

Trigger
Daily feature work
Who
Engineering · using Code Building Assistant ({dept}-builder-plugin) · four sub-personas inside (Dev · QA · BA · Reviewer)
Loop per feature
QA writes test cases first → Dev writes code following docs → BA updates docs → Reviewer checks tests + PR hygiene → merge to development → auto-deploy → regression
Typical iterations
Dozens of PRs across the project · merge cadence: multiple per day
Output
Working features · tests passing · regression green · deployed to dev
08

UAT iterations · client validation

Trigger
AM requests UAT after dev regression passes · UAT env auto-creates
Who
AM + Client · using the UAT env
Loop
Client validates → issues filed in GitHub Issues → fixed in dev → merged to UAT → re-validated
Notes
Optional on small projects — staging often serves as the client preview and UAT is skipped
Output
UAT sign-off as a signed git tag · e.g. v0.5-uat
09

Pre-go-live gate · the hard stop

Trigger
UAT signed (or dev signed if UAT skipped)
Who
DevOps + Security · using Quality & Delivery Assistant (ai-sdlc-devops-plugin)
What runs
VAPT · SAST · PenT · E2E finalisation · load test (peak × 1.5) · performance reports — all auto-generated · SLOs confirmed
Gate behaviour
If any check fails, the gate stays shut. No production deploy.
Output
Release Readiness Report signed by TL + QA + Security + DevOps + AM
10

Production deploy

Trigger
Gate passes · all sign-offs collected
Who
DevOps + AM
What happens
Production env provisions (first deploy) or rolling deploy (subsequent). 30-minute heightened monitoring window. Rollback rehearsed and runbook linked.
Output
{project}.dhwaniris.in (or client's own domain via CloudFront) · live to public users
11

Operate · always-on

Trigger
Production live · for the life of the application
Who
DevOps · using Production Monitoring Assistant (ai-sdlc-devops-plugin) · plus Mukhbir runtime health
Weekly
CPU < 50% check every 7 days · backup verification · health report to project owner
Alerts
Three-tier escalation — first alert info · second alert action · third alert auto-scales (up or down based on signal)
Monthly
Client-facing health report · learnings feed back to KM for the next project's Charter

The eleven stages are the same on every project. Their depth changes — a 5-day fit-gap might collapse Stages ⑥–⑦ into one merge and skip ⑧ entirely; a 4-week greenfield runs every stage with multiple iteration cycles. The gates (⑤ and ⑨) never get skipped.
