01 · How we ship · plugin architecture

Five Claude Code plugins. One hub. One shared skills repo.

Each persona gets a packaged set of skills, agents, and prompts as a Claude Code plugin. Each plugin lives in its own GitHub repo. One hub holds the SOPs and onboarding so anyone can find which plugin is for them and how to use it.

Pre-build · Stage A Steps 1–8

ai-sdlc-intent-plugin

Intent · Solutioning · Planning. Drives Stakeholder Map · Discovery · Charter · FRs · RBAC · Workflows · Data Model · Wireframes. Persona: Product Builder. Source docs already exist (Harshdeep + Nihaan, Session 1) — this plugin packages them as a Claude Code distribution.

Sign-off · Stage A Steps 9–10 + Stage C

ai-sdlc-architect-plugin

Architecture finalisation · validation before UAT · pre-go-live sign-off. Persona: Architect. Connects to ADR templates and the reference IaC catalogue. Built by Ankit + Deepak per Session 2.

Execute · Stage B

ai-sdlc-builder-plugin

Stage B execution. Master-prompt creation + execution command. Four personas inside: Developer · QA · BA · Reviewer. AI writes test cases first (from the PRD), then code; developer follows docs strictly, no extra scope; reviewer checks tests + docs + PR hygiene; BA owns comms + doc-as-you-go. Built by Ankit + Deepak.

Streams 3 + 4

ai-sdlc-devops-plugin

Pre-go-live gates: VAPT · SAST · PenT · E2E finalisation · load · perf reports — all auto-generated. No public users before this passes. Post-prod monitoring: weekly health · backup + utilisation reports · CPU < 50% check every 7 days · 3-tier alert escalation (3rd alert = critical = auto scale up/down). Persona: DevOps. Built by Ankit.

Shared infrastructure

ai-sdlc-skills

Central skills repo. Every *-author skill lives here — semver-tagged, eval-suite-gated. Plugins consume from here; projects pin versions in .claude/skills.lock.json. Prody's deck: "AI raises PRs into the skills repo when gaps appear."

Onboarding · SOPs · index

ai-sdlc-hub

Central reference site (GitHub Pages). Holds the SOPs, the per-persona onboarding paths, the plugin docs, the deployment lifecycle, the FAQs. One URL anyone in the team can land on and find the right plugin for their role.

Release sequence — Phase 0 to Phase 4

  1. Phase 0 · Foundation. ai-sdlc-hub + ai-sdlc-skills. The rails everyone else uses. Without these, every plugin reinvents structure.
  2. Phase 1 · Pre-build. ai-sdlc-intent-plugin. Stage A docs already exist — fastest plugin to ship.
  3. Phase 2 · Sign-off. ai-sdlc-architect-plugin. Makes Sol Doc reviews and pre-go-live sign-offs auditable.
  4. Phase 3 · Execute. ai-sdlc-builder-plugin. Four Builder personas live. AI writes tests + code together. Triggered when Sol Doc is signed and the first prompt fires.
  5. Phase 4 · Ship + Run. ai-sdlc-devops-plugin. Pre-go-live gates auto-run. Post-prod monitoring on every app.

Repo names are proposals. Naming follows existing Dhwani convention (kebab-case · ai-sdlc- prefix). Change any of them in the closing session — the structure is what matters, not the names.

05 · From wrapper to production

One repo. Five environments. One gate at each.

Builder pushes files into the GitHub wrapper. Environments self-provision per branch. DevOps owns the gate to production. No public users before the gate passes.

①

Wrapper · GitHub repo

Builder creates the project repo from the Dhwani starter (one command — see §06). If Builder & AM are the same persona, the request is self-service. Files arrive in GitHub; first prompt is ready to execute.

②

Staging · auto-provisioned

Spins up automatically when the repo is wrapped. The first AI prompt runs here. Auto-configured naming ({project}-staging).

③

Development branch → dev env

Merge to the development branch → auto-deploys to dev env. Regression runs here. Iterations land without ceremony.

④

UAT · often optional

Once development passes regression, UAT auto-creates. Client validates. Staging can also serve as client preview if a full UAT cycle is overkill on a small engagement.

⑤

Pre-go-live gate · DevOps plugin

VAPT · SAST · PenT · E2E finalisation · load · perf reports — all auto-generated. SLOs confirmed against the Sol Doc. If any check fails, the gate stays shut.

⑥

Production

Public users only after the gate passes. Post-deploy, weekly monitoring takes over: CPU < 50% check every 7 days · 3-tier alert escalation · 3rd critical alert triggers auto scale up / down.

06 · Adopt

08 · What we need from leadership

To deliver Streams 3 + 4, four quarters.

People

2 dedicated DevOps engineers · 4 quarters.

One anchoring Stage C (Kotwal integration · RRR pipeline · evidence flow). One anchoring Operate + Day 1 starter + the inter-stream bridge skills. Plus part-time support from each project's Tech Lead during their rollout window.

Tooling

Stage A is free. Stage B/C/Operate need licences.

Stage A: GitHub + Claude CLI + Mermaid + Stitch + Fireflies — all free or already-licensed (Prody's stack). Stage B/C/Operate: a small set of paid licences for security scanning, observability, status pages. Kotwal + Mukhbir anchor the internal layer.

Decisions

Five standards calls only leadership can make.

  • Front-end stack — Vue or React, both, or stack-dependent?
  • CI/CD platform — standardise on GitHub Actions across all stacks?
  • Cloud target — AWS primary, on-prem for govt?
  • Env portal — Backstage (OSS) or paid (Port / Cortex)?
  • Project tiering — who classifies day-to-day?
