AI-SDLC · Stream 4 — DevOps + Observability

How DevOps supports the AI-SDLC at Dhwani RIS

The AI-SDLC at Dhwani runs as five parallel streams. This is the input from Stream 4 — DevOps + Observability: how DevOps plugs into Stage A (Requirement Gathering — pitched today), how it supports Stages B and C when they ship, and what it owns end-to-end across the always-on Operate layer.

Prepared by Dhwani RIS DevOps Aligns with Prody's Stage A pitch · 14 May 2026 Status Draft v0.3 · 2026-05-14

00 Where this document fits

The AI-SDLC at Dhwani has been scoped over the 7 May and 12 May sessions. On 14 May (Closing Session 2/2), Prody's Group PM Office is pitching Stage A · Requirement Gathering. That deck is the authoritative source — its terminology, its 10 RG steps, its 3 phases, its tool philosophy, and its 4-stack overlay are the framework we operate within.

This document is the DevOps vertical's input. It reuses Prody's exact step names and skills. It positions DevOps as Stream 4 (parallel to RG, Dev+Testing, Testing+Perf, and KM) and as one of the three always-on verticals (KM · Project Governance · DevOps) that check every phase.

What DevOps adds on top of Prody's framework. Prody's pitch defines what the framework is. DevOps owns how it gets adopted (one-command starter — see §09), how it evolves (raise mechanism + Champion + bi-weekly review — see §10), and how leadership measures it (monthly telemetry — see §10). Three loops, owned by Stream 4.

Agent governance — DevOps-owned

  1. Catalogue. Every agent in production use is registered with owner · version · scopes. New agents are reviewed before wide adoption.
  2. Where it runs. Local agents are at the persona's discretion. CI-invoked agents come only from approved workflows.
  3. Access scope. No production secrets, raw PII, or push to protected branches. Vault-injected credentials only.
  4. Output review. Every agent output that lands in code or infra passes the same gates as human output — PR review, CI, ai-generated label.
  5. Cost tracking. API spend per project tracked monthly. Budget cap per project tier.
  6. Audit trail. CI agent invocations logged (input + output) for 90 days. Allows incident replay.
  7. Versioning. Agent prompts and configs versioned in git. Any past run is reproducible.
  8. Deprecation. Agents not invoked in 90 days are quarantined and removed.

10 Evolve — how the framework stays alive

Static methodology gets stale. The Dhwani AI-SDLC has a built-in raise mechanism so anyone using the framework can shape it. DevOps owns the pipework. Three loops.

Loop 1 · Skills

The Dhwani Skills Repo

dhwani-ris/ai-sdlc-skills (to be created)

One repo · all skills · semver-tagged. Every skill carries owner, version, change log, and an eval suite. Projects pin versions in .claude/skills.lock.json.

Prody's deck: "AI raises PRs into the skills repo when gaps appear." DevOps wires that up: a project Issue tagged raise:skill mirrors to the skills repo; prompt-improver drafts a PR within 24h; the eval suite gates merge.

Loop 2 · Raises

How anyone surfaces a gap

  1. Hit a missing template, skill, or scaffold → open an Issue with the Raise template (PB, AR, AM, anyone).
  2. Champion triages within 48h. Label is accept / clarify / decline, with a one-line reason.
  3. Accepted raises enter the backlog. Sprint-sized: 2-week SLA. Bigger: ships in the next framework version.
  4. Major changes get a version bump (v1.x → v1.x+1) with a migration note.

Loop 3 · Review

Bi-weekly framework review + Champion

13 What we need from leadership

Three categories of support to deliver Stream 4 on the timeline above.

People

2 dedicated DevOps engineers, for four quarters.

DevOps owns two streams (3 and 4) plus the always-on vertical role. Two engineers covers: one anchoring Stage C (Testing & Performance, including Kotwal integration and the RRR pipeline) and one anchoring the Operate layer + the Day 1 starter + bridge skills. Plus part-time support from each project's Tech Lead during their rollout window.

Tooling

Stage A is free. Stage B/C/Operate need a small set of licences.

The RG stack is GitHub + Claude CLI + Mermaid + Stitch + Fireflies — all free or already-licensed. Stage B/C/Operate need security scanning (Semgrep), observability, status page, paid env portal (or Backstage). Kotwal + Mukhbir anchor the internal layer.

Decisions

Standards calls only leadership can make.

  • Front-end stack within the Frappe / Non-Frappe stacks — Vue vs. React?
  • CI/CD platform — standardise on GitHub Actions across all four stacks?
  • Cloud target — AWS primary, on-prem for government projects?
  • Build vs. buy the env portal — Backstage (OSS) or paid (Port / Cortex)?
  • Who classifies project criticality tiers?
