DevOps Strategy · POC

The AI-SDLC DevOps Playbook

Operationalising AI-assisted delivery at Dhwani RIS — from commit to production, safely and at speed. Empowering Social Impact Through Technology, with the guardrails to match.

Owner Ravi Jangir · DevOps Audience Tech Leads · QA · PM · Security Version Draft v0.1 · 2026-05-14

00 Why this exists

Code now ships from analysts, PMs, and developers — all assisted by AI. Traditional linting and semantic checks were designed for human-authored code with predictable failure modes. AI-authored code fails differently: plausible-but-wrong logic, hallucinated APIs, generic boilerplate that hides edge-case gaps, and mock-heavy tests that pass without exercising behaviour.

The DevOps layer at Dhwani RIS must adapt. We need to guard the codebase from new failure modes, accelerate environment access so contributors aren't blocked, gate releases against a complete readiness checklist, and keep production observable enough that we know — before our clients do — when something drifts.

The shift in one line. Old approach: assume a human wrote it, check syntax and style. New approach: assume an LLM may have written it, verify intent, behaviour, and absence of plausible nonsense.

P1 AI-Aware Code Analysis

Linters check syntax, not intent. The pipeline below assumes any commit may contain AI-generated code and adds new gates specifically tuned to LLM failure modes.

New AI-specific checks

  • Suspicion gate. Flag commits where new functions look generic versus project style — comments that narrate the obvious, abstract base classes added for a single concrete usage, over-defensive null guards in trusted code paths.
  • Test quality gate. Reject PRs where more than 70% of assertions are against mocked return values. Tests that don't exercise real behaviour are worse than no tests because they signal false confidence.
  • Hallucination check. Validate that imported symbols actually exist in the declared version of the dependency. Catches LLM-invented APIs and renamed functions in older library versions.
  • Dependency justification. Any new dependency added in an AI-assisted PR requires a one-line "why" in the PR description, enforced by a CI check.
  • PR labelling. ai-generated or ai-assisted labels are required; reviewers scrutinise logic, not just style, on labelled PRs.

Pre-commit baseline (project template)

  • Python — black, isort, autoflake, flake8, mypy, bandit, safety
  • JS / TS — prettier, eslint, tsc --noEmit
  • Cross-cutting — commitlint (conventional commits), gitleaks / detect-secrets, actionlint for workflows
  • Frappe-specific — custom hooks for ignore_permissions=True, allow_guest=True, raw SQL via f-string

P2 Self-Service Environments

Any team member should be able to request a staging or UAT environment in under fifteen minutes, without DevOps in the loop. Environments are tiered so teams pick the right size for the job.

Architecture

  • IaC. Terraform modules per service (building on bse-iaas); environments are parameterised stacks of the same modules.
  • Orchestration. GitHub Actions workflow_dispatch triggered from a small internal portal (Backstage candidate, or a plain Next.js form against the Actions API).
  • DNS. <project>-<branch-or-purpose>.staging.dhwaniris.in.
  • Data. Per-env Postgres, seeded nightly from an anonymised prod snapshot. PII fields (Aadhaar, PAN, phone) hashed at snapshot time.
  • Secrets. Vault or AWS Secrets Manager, injected at boot — never in env files.
  • Lifecycle. Auto-destroy after 7 days idle; 24h warning; one-click extend.

P3 Pre-Go-Live Gate

A release-readiness portal. Teams request promotion → the portal triggers required scans → only when every report passes (or has a signed exception) is Promote to Production enabled. Kotwal anchors the security side; the rest of the checklist covers correctness, performance, and operability.

P4 Post-Deployment Ops

Observability is not optional and not bespoke. Every project starts with the same default-on stack and the same SLO budgets, overridable where the project demands it. Reports go out on a schedule the team picks — weekly default, monthly for stable apps.

Default observability stack

  • Metrics — Prometheus + Grafana
  • Logs — Loki + Grafana
  • Traces — Tempo + Grafana
  • Runtime health — Mukhbir (existing internal tool)
  • Alerts — SLO burn-rate to Slack + PagerDuty
  • Status page — per project, public or private

02 Roadmap (16 weeks)

  1. Phase 0 · Weeks 0–1

    Inventory

    Tooling audit. Project tiering. SLO baseline per project. Identify the pilot project.

  2. Phase 1 · Weeks 1–4

    AI-aware CI

    Pre-commit template rolled out. AI-suspicion gates wired in. Kotwal integrated as a required CI check on PRs.

  3. Phase 2 · Weeks 4–8

    Self-service environments

    Tier S and Tier M live for the pilot project. Portal MVP — a form, an Actions trigger, a URL back.

  4. Phase 3 · Weeks 8–12

    Pre-go-live gate

    Portal release-readiness flow. All 12 checks wired. First release passed through the gate end-to-end.

  5. Phase 4 · Weeks 12–16

    Ops reports

    Observability template applied to all in-flight projects. Auto-report generator delivering weekly + monthly. Status pages live.

04 Open Questions for the Team

  1. CI/CD platform — standardise on GitHub Actions across all projects, or accept mixed tooling?
  2. Cloud target — AWS primary? On-prem for some government projects? How do we treat hybrid?
  3. Kotwal — what is the API / CLI surface for CI integration? Versioning policy?
  4. Portal — build (Backstage) or buy (Port, Cortex)? What's the support cost over 12 months?
  5. Project tiering — who owns the classification: PM, Tech Lead, or DevOps?
  6. Government-project specifics — data residency requirements per state and per scheme?
  7. Exception process — who signs exceptions on failing pre-go-live checks, and how is it audited?
